# Epic 0001: Shareable Layouts

## Goal

Consolidate duplicated UI boilerplate across app surfaces so Blog Engine, Radar, Hyperlocal, and eventually Prompt Studio share common layout, navigation, page, modal, and onboarding presentation primitives.

## Context

The current UI has several product-specific component trees that repeat the same structure with small variations. See `TECH_DEBT.md` for the initial duplication audit.

Primary duplication areas:

- App shell wrappers in `app/apps/*/layout-client.tsx`.
- Product headers in `components/blog-engine`, `components/radar`, and `components/hyperlocal`.
- Blog Engine and Radar welcome screens.
- Dashboard page frames, cards, empty states, and status banners.
- Blog Engine and Radar upgrade modals.
- Onboarding chat presentation for Blog Engine, Radar, and Hyperlocal.

## Non-Goals

- Do not redesign the apps.
- Do not merge unrelated product-specific business logic.
- Do not force Prompt Studio into the new shell until the sidebar/chat header contract is understood.
- Do not change route semantics or subscription behavior while extracting UI primitives.

## Architecture Direction

- Prefer data/config-driven shared components with app-specific slots.
- Keep product-specific wrappers thin and readable.
- Extract presentation before extracting state machines.
- Keep each issue small enough to validate independently.

## Done When

- Shared primitives cover the repeated UI paths without obscuring app-specific behavior.
- Existing routes continue to render the same app experiences.
- New shared components have clear props and are easy to reuse for future app surfaces.
- The duplication called out in `TECH_DEBT.md` is either resolved or explicitly left for a future epic.
